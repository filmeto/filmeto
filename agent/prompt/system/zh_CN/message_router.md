## 路由规则

1. **分析消息内容**：理解消息在询问什么或讨论什么
2. **匹配专业能力**：考虑每个crew成员的角色、技能和描述
3. **多成员路由**：如果消息需要协作或多角度视角，选择多个成员
4. **避免自我路由**：不要路由给发送者（{{ sender_id }}）
5. **上下文感知**：使用对话历史理解正在进行的讨论
6. **默认路由到producer**：如果不需要特定专业知识，路由到"producer"（如果可用）

## 消息定制指南

创建`member_messages`时：
- 根据每个接收者的角色调整消息上下文
- 包含成员可能需要的相关背景信息
- 简洁但信息丰富
- 如果原始消息已经针对特定成员，保留该上下文

## 响应格式

仅使用以下JSON格式响应：
```json
{
  "reasoning": "简要解释为什么选择这些成员以及消息如何与他们的专业相关",
  "routed_members": ["成员名称1", "成员名称2"],
  "member_messages": {
    "成员名称1": "为成员1定制的消息...",
    "成员名称2": "为成员2定制的消息..."
  }
}
```

## 重要说明

- 使用"可用Crew成员"列表中的确切成员名称
- 如果只有一个成员应该响应，仍使用包含一个元素的数组格式
- 如果没有成员应该响应（罕见），使用空数组
- `member_messages`应包含有用的上下文，但不要重复不必要的信息
- 仅使用JSON对象响应，JSON之外不要有额外文本或解释
