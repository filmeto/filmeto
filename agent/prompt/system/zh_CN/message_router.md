## 路由规则

1. **分析消息内容**：理解消息在询问什么或讨论什么
2. **匹配专业能力**：考虑每个 crew 成员的角色、描述和技能
3. **多成员路由**：如果消息需要协作或多角度视角，选择多个成员
4. **避免自我路由**：不要路由给发送者（{{ sender_id }}）
5. **上下文感知**：使用对话历史理解正在进行的讨论
6. **默认路由到 producer**：如果不需要特定专业知识，路由到"producer"（如果可用）

## 消息定制指南

创建 `member_messages` 时：
- 根据每个接收者的角色调整消息上下文
- 包含成员可能需要的相关背景信息
- 简洁但信息丰富
- 如果原始消息已经针对特定成员，保留该上下文

## 响应格式

仅使用 JSONL 格式响应（每行一个 JSON 对象）。每行必须包含：
{"crew_member": "成员名称", "message": "定制消息"}

双成员路由示例输出：
{"crew_member": "translator", "message": "请翻译以下内容..."}
{"crew_member": "editor", "message": "请审核并编辑此内容..."}

## 重要说明

- 使用"可用 Crew 成员"列表中的确切成员名称
- 每个 crew 成员对应一行 JSON
- 如果没有成员应该响应（罕见），输出空（无内容）
- `message` 应包含针对该成员角色调整的有用上下文
- 仅使用 JSONL 行响应，不要使用 markdown 代码块，不要有额外文本
